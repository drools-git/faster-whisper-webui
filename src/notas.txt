    ow I can see exactly what's happening. The issue is auto_parallel: true in your config.json5. Here's the chain of events:
Because auto_parallel is true, the app configures 8 CPU worker processes + 1 GPU process (line 159 in the terminal: Using GPU devices ['0'] and 8 CPU cores)
When you submit a URL, it downloads the audio (likely works fine)
Then since _has_parallel_devices() returns true, it takes the parallel processing path in vadParallel.py
This calls multiprocessing.get_context('spawn') (line 46 of vadParallel.py) and creates a pool of 8 processes for VAD and another pool for GPU transcription
On Windows, spawn multiprocessing means each subprocess has to re-import every module, reinitialize CUDA, and reload both the Silero-VAD model and the faster-whisper model from scratch
This is almost certainly deadlocking -- the spawned child processes hang during initialization, and print() output from child processes doesn't appear in the parent console
For a 1-minute audio file with a single GPU, parallel processing is completely unnecessary and actually adds enormous overhead.
The fix: Change auto_parallel from true to false in config.json5 (line 81). Switch to Agent mode and I can make this change for you, or edit it manually:
